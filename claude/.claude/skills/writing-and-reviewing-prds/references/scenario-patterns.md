# 场景化 PRD 写法

仅在需求匹配对应场景时读取和使用。不得把资金、接口或移动端专用章节机械套入无关需求。

## 使用规则

1. 先执行 `SKILL.md` 的现状确认门禁，再使用本文件模板。
2. 使用业务实际名称和用户提供的字段，不复制其他项目的产品、渠道、资方或系统名称。
3. 写“逻辑同 X”“逻辑同原有”“其他字段不变”前必须能够读取对应现有材料，并明确哪些逻辑相同、哪些不同；无法读取时继续追问。
4. 领域模板用于补充通用 PRD，不替代业务场景、异常和验收标准。

## 1. 管理后台、资金与接口类

### 1.1 后台列表页

列表页必须同时包含字段、筛选和操作，即使操作只有查询、重置或刷新也不能省略。权限需要拆成页面权限、操作权限和数据权限。页面展示总条数、总金额等汇总值时，必须说明是否基于当前筛选结果以及统计口径。

### 1.2 接口或三方系统对接

| 项目 | 必须说明的内容 |
|---|---|
| 接口名称与方向 | 调用方、被调用方、同步或异步 |
| 调用时机 | 由哪个业务动作或系统事件触发，触发前置状态 |
| 请求数据 | 字段、来源、必填、格式、加密或脱敏要求 |
| 成功判断 | 返回字段及其组合条件，不得只写“返回成功” |
| 失败处理 | 错误码、用户提示、业务状态、是否允许重试 |
| 异步回调 | 回调地址、验签、重复回调、乱序回调和最终状态 |
| 幂等与重试 | 幂等键、超时、重试次数或策略、人工补偿入口 |
| 操作留痕 | 请求编号、业务编号、时间、结果、错误信息和关联日志 |

存在多接口顺序、回调或补偿路径时增加时序图，并在图后说明正常路径和兜底路径。

### 1.3 两套账、对账与资金处理

涉及平台账与资金方账、业务账与财务账或其他两套口径时，分别说明：

1）每套账的记账主体、金额、时间和状态。

2）请款、放款、还款、退款、代偿、回购、冲正等事件分别如何影响两套账。

3）先后顺序、接口调用、异步回调和失败补偿。

4）对账文件来源、匹配键、差异分类、处理责任人和结清条件。

5）在途、重复、部分成功和跨日处理方式。

用时序图展示资金或账务先后关系；正文必须保留金额和状态规则。

### 1.4 协议与合同

| 项目 | 必须说明的内容 |
|---|---|
| 协议名称与版本 | 完整名称、生效版本、版本日期 |
| 签署方式 | 阅读确认、主动签署、默签或其他方式 |
| 签署时机 | 哪个流程节点触发，失败是否阻塞主流程 |
| 合同编号 | 组成字段、位数、序号和唯一性规则 |
| 模板与变量 | 模板来源、变量来源、缺失变量处理 |
| 存储与查询 | 文件存储、查看入口、下载和权限 |
| 更新兼容 | 新旧版本适用范围和历史合同处理 |

协议、凭证或合同从一个页面迁移到另一个页面时，还要说明：原入口是否删除、新入口出现条件、返回路径、最新版本如何选择、不同签署主体如何适配、历史文件是否继续展示。

存在“空模板 → 已填写 → 已签署或签章 → 传输 → 展示或下载”等阶段时，分别说明每个阶段的生成时机、文件内容、存储位置、传输对象、失败处理和是否阻塞主流程。

### 1.5 数字范围与边界

数字区间必须明确数据类型、单位、最小值、最大值、是否含边界、空值含义、无穷值表达、判断时机和不满足时的处理。不要只写“在范围内”。

### 1.6 批量任务与异步任务

说明任务创建、文件或数据范围、最大数量、排队、执行中、部分成功、全部成功、失败、取消、重试、结果下载、通知和过期清理。复杂路径增加时序图。

批量导入还必须说明：

1）文件类型、模板版本、工作表、列名、顺序、格式和示例。

2）文件级校验和逐行校验，以及重复数据、空值和非法值处理。

3）任务状态及每种状态的进入和退出条件。

4）成功行、失败行、失败原因、结果文件和下载入口。

5）部分成功是否提交、重新上传是否幂等、重复任务如何识别。

同一业务同时提供单笔和批量操作时，核心校验、状态变化、权限和副作用必须一致；差异点需要单独说明。

### 1.7 统计与指标口径

每个指标必须说明统计对象、时间字段、状态范围、金额字段、去重键、退款或退费处理、时区、刷新频率和空值处理。相似指标存在不同口径时并列对比，不能用同一个名称模糊处理。

### 1.8 适用范围、配置和唯一性

新增规则只对部分产品、渠道、组织、资金方或业务类型生效时，明确适用矩阵、默认值、配置入口、配置权限、生效时间和未配置时的行为。

多个维度共同确定一个方案或规则时，列出组合唯一键、重复配置校验、优先级和冲突处理。为未来范围预留配置时，仍要明确本期启用范围，不能只写“预留口子”。

### 1.9 金额与计算规则

每条公式必须定义：

1）计算变量及其业务含义、来源、单位和空值。

2）计算时点、日期字段、账期和跨日口径。

3）公式、先后顺序、精度、舍入方式和最终展示格式。

4）区间边界、阈值判断和等于阈值时的处理。

5）累计金额、部分收款或多次操作的汇总范围。

6）人工修改、减免或回滚后哪些值重新计算，哪些保留历史。

不同产品、计息方式或时间区间采用不同公式时，使用决策表并列展示，禁止将多个分支塞进一段话。

### 1.10 撤销、逻辑删除与状态回退

取消、撤销或删除业务记录时，明确是物理删除还是逻辑删除；原状态回退到哪里；账单、请求、任务和下游数据如何处理；历史记录从哪里查看；是否允许再次发起；失败时如何恢复。

## 2. 移动端或 C 端

### 2.1 入口、触发、路径与状态

按以下顺序说明：

1）入口：扫码、菜单、消息、工作台、外部链接或其他入口。

2）触发：触发角色、时间、事件和携带参数。

3）前置判断：登录、授权、关注、白名单、客户状态或业务资格。

4）页面路径：进入页面、跳转顺序、返回行为和参数传递。

5）页面状态：加载、默认、空数据、处理中、成功、失败、无权限和失效。

6）异常引导：提示文案、可执行动作、重试或联系客服方式。

涉及扫码、授权、进件、支付或推送链路时增加端到端流程图。

### 2.2 字段、脱敏与复制

| 项目 | 必须说明的内容 |
|---|---|
| 字段展示 | 来源、格式、空值、不同状态下的展示 |
| 脱敏 | 脱敏角色、规则、可否查看完整值 |
| 点击与复制 | 可点击区域、复制内容、成功提示和权限 |
| 输入 | 键盘类型、长度、格式、必填、校验和错误提示 |

### 2.3 完整文案与通知

通知、按钮、弹窗、错误提示和推送消息必须给出完整字面量，并说明变量、触发时机、发送对象、渠道、跳转目标、重复发送、失败补偿和过期时间。不得只写“参考原文案”或“提示失败”。

### 2.4 状态与视觉

需要颜色区分状态时，列出“业务状态 → 展示文案 → 颜色或样式”的对应关系。颜色值以设计稿或现有设计规范为准；缺少依据时要求补充，不得套用其他项目的固定 RGB。

### 2.5 权限

内部移动工作台同时说明菜单权限、操作权限和数据权限；面向客户的页面说明身份校验、数据归属、链接转发后的访问控制和敏感信息保护。

## 3. 禅道式或简版需求

用户明确要求简版、禅道需求或直接进入需求详情时，可以采用以下密度：

1）标题包含主题、重要性和期望时间（如已确认）。

2）用一句话交代业务场景和目标，随后进入页面、流程或规则。

3）保留所有关键字段、触发点、状态、权限、异常、接口和验收标准。

4）追加修改时记录更新日期、修改条目和影响范围；重要改动继续红色加粗。

简版只减少铺垫，不减少业务事实。

## 4. 配图选择

| 内容 | 推荐图形 | 必须说明 |
|---|---|---|
| 跨系统或端到端流程 | 流程图或时序图 | 起点、判断、系统交接和终点 |
| 非顺序状态流转 | 状态机 | 触发点、条件、终止态和回退 |
| 接口、回调或两套账 | 时序图 | 调用方向、先后顺序、超时和补偿 |
| 菜单、Tab、卡片层级 | 层级图或截图 | 层级、入口、显示条件和跳转 |
| 批量或异步任务 | 时序图或流程图 | 状态、部分成功、重试和通知 |

每张图后增加 `> 图说：`，用一段话概括正常路径、关键判断和异常兜底。图中的字段、状态、角色和接口方向必须与正文一致。
