# PRD 校验清单

## 目录

1. 业务理解与技术可读性；2. 现状确认；3. 业务流程；4. 状态机；5. 页面与字段；6. 现有与新逻辑；7. 重要改动；8. 后台规则；9. 权限、数据与留痕；10. 异常与示例；11. 视觉材料；12. 验收与 AI 可读性；13. 数据、接口与非功能；14. 评审、答疑与同步。

## 校验结论等级

| 结论 | 含义 |
|---|---|
| 完整 | 所有适用的事实、决策、规则、依据和验收标准均已确认 |
| 有条件可用 | 仍有不阻塞实施的缺失项，并且已经明确记录 |
| 阻塞 | 相关现有逻辑或其他高影响业务事实尚未确认 |

不得只给评分。每个问题必须关联依据、需求编号或文档位置。

## 1. 业务理解

- [ ] 已明确业务用户和角色。
- [ ] 已明确时间、情境、操作、期望结果和风险。
- [ ] 问题、目标、成功标准、范围和非目标相互一致。
- [ ] 已定义业务对象及其归属。
- [ ] 原型、设计稿、接口、合同和附件等关联材料已集中列出并可访问。

### 技术可读性与术语

- [ ] 一页核心摘要能够说明目标、范围、核心改动、影响系统和阻塞问题。
- [ ] 技术人员可以通过需求编号从摘要定位到详细规则、图示和验收标准。
- [ ] 核心规则没有淹没在长段落、重复背景、截图批注或会议记录中。
- [ ] 一段只表达一个核心规则，重复属性或映射已经改用表格。
- [ ] 可能产生歧义的业务对象、状态、时间、金额、角色和系统已经进入术语表。
- [ ] 全文遵守“一概念一名称、一名称一含义”，历史别名只在术语表中映射。

### 编号与可读性

- [ ] 编号层级统一为 `1、` → `1）` → `⑴` → `①`，没有同层级混用。
- [ ] `1）2）3）` 和 `⑴⑵⑶` 等编号项均独占一行。
- [ ] 长子项已经分行并缩进，没有多个规则挤成一整段。
- [ ] 完整文案、字段值或用户可见提示使用独立引用块或清晰区块展示。

## 2. 现状确认

每项现有系统改动都必须检查本章节。

- [ ] 已明确系统、模块、菜单、页面、入口或后台任务。
- [ ] 已确认现有用户、角色、权限和数据范围。
- [ ] 已确认现有流程、状态、流转、触发点、条件和副作用。
- [ ] 已确认现有可见字段、取值来源、计算、映射、操作和校验。
- [ ] 已确认现有隐藏规则、系统集成、通知、操作留痕和异常处理。
- [ ] 每项现有事实都有依据来源。
- [ ] 没有把截图当作隐藏逻辑的证明。
- [ ] 未知现有逻辑已进入现状待确认清单。

以上任一适用项不满足，依赖该项的最终需求必须标记为阻塞。

## 3. 业务流程

- [ ] 每个节点都有明确的操作人或系统。
- [ ] 入口条件、触发点、处理过程、结果和失败路径清晰。
- [ ] 分支条件可以明确区分。
- [ ] 跨角色、跨系统交接明确责任人。
- [ ] 流程图与文字描述一致。
- [ ] 流程图后有 `> 图说：`，能够概括起点、关键判断、结果和异常分支。
- [ ] 多条规则可能同时命中时，已明确规则优先级、判断时机、短路条件和互斥范围。
- [ ] 存在继续、重办、恢复或重建分支时，已明确数据带出、保留、清空和旧记录处理。
- [ ] 跨角色流程使用泳道图或明确标注角色的流程图，不依赖无主语描述。
- [ ] 多条件组合使用判定表或判定树，能够看出条件、优先级、结果和兜底。

## 4. 状态机

- [ ] 每个状态的含义唯一且清晰。
- [ ] 每次流转都有原状态、触发点、条件、操作人和新状态。
- [ ] 已明确库存、额度、账务、通知、待办、操作留痕或系统集成等副作用。
- [ ] 已处理非法、重复、并发、取消、冲正和失败流转。
- [ ] 状态图与流转表一致。

## 5. 页面与字段

- [ ] 每个页面都有位置、用途、用户、权限和视觉材料编号。
- [ ] 已说明加载、空数据、异常和无权限状态。
- [ ] 每个字段都有位置、来源、计算或映射、格式、空值处理和权限。
- [ ] 字段说明包含适用的可编辑、只读、默认值、动态展示条件和所属业务对象。
- [ ] 已明确适用的筛选、默认值、排序、分页、导出和点击行为。
- [ ] 每个操作都有可见条件、可用条件、二次确认、系统处理、成功结果和失败结果。
- [ ] 重要操作说明状态、数据、下游接口、通知、留痕和失败回滚等副作用。

### 管理后台或内部管理系统专项检查

- [ ] 每个适用页面均按“列表页字段 → 筛选项 → 操作 → 权限 → 分页 → 其他页面元素”的顺序编写。
- [ ] 列表页字段不只列名称，还说明取值来源、计算或映射、格式、空值、权限和点击行为。
- [ ] 筛选项说明类型、默认值、选项来源、查询逻辑和重置行为；日期筛选说明边界与时区。
- [ ] 操作按钮说明可见角色、可用条件、处理逻辑、成功结果和失败结果。
- [ ] 权限同时覆盖页面可见、按钮可见、按钮可用和数据范围。
- [ ] 分页说明每页数量选项 `10/20/50/100`、默认值、页码处理、总数展示和跨页勾选规则。
- [ ] 其他页面元素覆盖适用的排序、批量选择、空状态、加载状态、异常状态和跳转行为。
- [ ] 总条数、总金额等汇总值说明是否基于当前筛选结果及具体统计口径。
- [ ] 宽列表说明固定列、操作列、列宽、横向滚动和超长内容处理。

### 移动端或 C 端专项检查

- [ ] 已说明入口、触发方式、携带参数和前置判断。
- [ ] 页面路径包含进入、跳转、返回和参数传递。
- [ ] 字段规则包含适用的脱敏、点击复制、输入校验和错误提示。
- [ ] 页面状态覆盖加载、默认、空数据、处理中、成功、失败、无权限和失效。
- [ ] 通知、按钮、弹窗、错误提示和推送给出完整文案，而不是只写“参考原文案”。
- [ ] 内部移动工作台覆盖菜单、操作和数据权限；客户页面覆盖身份与数据归属校验。

### 资金、接口或三方系统专项检查

- [ ] 已读取 `scenario-patterns.md` 中匹配的场景模板。
- [ ] 接口说明包含方向、调用时机、请求数据、成功条件、失败处理、回调、幂等和操作留痕。
- [ ] 存在两套账时，分别说明记账主体、事件影响、先后顺序、对账和差异处理。
- [ ] 协议或合同说明版本、签署方式、签署时机、编号、模板变量、存储和新旧兼容。
- [ ] 协议、凭证或合同发生页面迁移时，已说明原入口、新入口、主体适配和历史文件展示。
- [ ] 文件存在模板、填写、签署或签章、传输、展示等阶段时，已说明各阶段生命周期。
- [ ] 数字范围明确单位、边界、是否包含、空值含义和不满足时的处理。
- [ ] 统计指标明确对象、时间字段、状态、金额字段、去重、退款处理、时区和刷新频率。
- [ ] 适用范围说明产品、渠道、组织等维度，并明确配置入口、默认值、生效时间和组合唯一性。
- [ ] 金额公式定义变量、来源、单位、日期口径、精度、舍入、阈值、累计范围和重算规则。
- [ ] 批量导入说明文件结构、逐行校验、任务状态、失败明细、结果下载和幂等。
- [ ] 单笔与批量操作的核心校验、状态、权限和副作用一致，差异已经单独说明。
- [ ] 撤销、取消或删除说明逻辑删除、状态回退、下游处理和历史记录。

## 6. 现有逻辑与新逻辑

- [ ] 现有逻辑与新逻辑已经明显区分。
- [ ] 每项改动都说明原因和影响范围。
- [ ] 删除的行为已经明确说明，而不是静默省略。
- [ ] 为避免误解，保留了必要的相关未改动逻辑。
- [ ] 逻辑变化时已处理历史数据和进行中的记录。
- [ ] 没有在基础逻辑未确认时直接写“其他字段不变”“逻辑同原有”或“沿用原逻辑”。

## 7. 重要改动展示

- [ ] 影响业务流程、状态、字段取值、权限、系统规则、数据结果或验收结论的重要改动已经红色加粗。
- [ ] Markdown 使用 `<span style="color:#FF0000"><strong>重要改动内容</strong></span>`。
- [ ] Word 和飞书文档使用实际红色字体与粗体样式。
- [ ] 只标记关键变化，没有把整段、整页或未变化内容全部标红。

## 8. 后台规则

- [ ] 每条规则都有编号、触发点、输入、处理、输出和副作用。
- [ ] 已区分现有规则和新规则。
- [ ] 已处理适用的重试、幂等、超时、回滚、补偿和人工恢复。
- [ ] 定时任务和外部事件说明执行时间及重复事件处理方式。

## 9. 权限、数据与操作留痕

- [ ] 查看权限与操作权限分开说明。
- [ ] 已明确组织、区域、客户、负责人等数据范围。
- [ ] 敏感字段说明脱敏和导出规则。
- [ ] 已明确必须记录的操作留痕。
- [ ] 数据来源、归属、时效和一致性要求清晰。
- [ ] 数据改为关联其他业务对象时，已说明唯一关联键、存量迁移、历史兼容和异常处理。

## 10. 异常与示例

- [ ] 主流程示例使用具体数值。
- [ ] 容易误解的规则包含边界或失败示例。
- [ ] 示例与公式、状态、权限和流程规则一致。
- [ ] 必要时说明用户可见的错误提示或处理结果。
- [ ] 已扫描适用的空态、弱网、断网、超时、并发、重复请求、权限拦截、部分成功、回滚和人工恢复。
- [ ] 异常场景说明系统状态、数据结果、用户提示、可执行动作以及日志或告警。

## 11. 视觉材料

- [ ] 现有截图、原型图和 UI 设计具有稳定编号。
- [ ] 文字需求引用对应视觉编号。
- [ ] 已标注新增、修改和删除元素。
- [ ] 可以识别视觉材料的版本或来源。
- [ ] 仅靠文字容易误解时，已要求补充视觉材料。
- [ ] 跨系统流程、状态机、接口或两套账时序、页面层级、批量异步任务在复杂时已经配图。
- [ ] 每张图都有 `> 图说：`，说明正常路径、关键判断和异常兜底。
- [ ] 图中的状态值、字段名、角色、接口方向和触发条件与正文一致。
- [ ] 图或正文修改时，另一方已经同步修改。
- [ ] 表达形式匹配信息类型：字段映射用表格、生命周期用状态机、复杂条件用判定结构、异步交互用时序图。

## 12. 验收与 AI 可读性

- [ ] 每项需求都有稳定编号。
- [ ] 验收标准说明前提、操作和预期结果，或采用等价的可验证规则。
- [ ] “快速”“友好”“大量”等主观词已经量化或删除。
- [ ] 已确认事实、业务决策、暂时假设和待确认问题相互分开。
- [ ] 交叉引用使用一致编号。
- [ ] 最终正文中没有隐藏未解决的阻塞问题。
- [ ] 追加修改记录了更新日期、修改条目和影响范围，重要改动仍然红色加粗。

## 13. 数据、接口与非功能要求

- [ ] 数据对象说明来源、归属、唯一键、时效、一致性和失败处理。
- [ ] 接口说明方向、时机、输入输出、成功条件、错误、超时、重试、幂等、回调和留痕。
- [ ] 只写与需求直接相关的非功能要求，并使用可验证的性能、容量、可用性、安全、监控或兼容性指标。
- [ ] 每项非功能要求都有目标值、验证方式和不满足时的处理。

## 14. 评审、答疑与交付物同步

- [ ] 跨系统、跨团队、状态复杂、资金账务、安全合规、数据迁移或高风险需求已经留下评审记录。
- [ ] 评审结论、否决方案、影响范围、负责人和截止时间已记录。
- [ ] 未解决问题有明确答疑负责人、回复渠道、截止时间和状态。
- [ ] PRD、原型、UI、接口文档和测试用例的版本及同步状态清晰。
- [ ] 原型、UI 或接口修改后，正文和验收标准已经同步；正文修改时也反向检查其他交付物。
- [ ] 更新后已高亮重要改动，并明确需要通知的开发、测试、设计、业务或运营角色。

## 必须采用的校验输出结构

```markdown
## Review conclusion
Complete / Conditionally usable / Blocked

## Confirmed content to preserve
- [Requirement ID or location]: [evidence]

## Blocking current-state questions
1. [One focused question and why it blocks the change]

## Missing requirements
| Location / ID | Missing content | Impact | Recommended action |
|---|---|---|---|

## Ambiguities and contradictions
| Location / ID | Problem | Competing interpretations | Required decision |
|---|---|---|---|

## Recommended edits
| Location / ID | Current text | Proposed change | Basis |
|---|---|---|---|

## Visual evidence required
| Related ID | Needed artifact or annotation | Reason |
|---|---|---|
```
