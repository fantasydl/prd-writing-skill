# PRD 清晰度与协同

复杂 PRD 的价值不在于篇幅，而在于技术人员能否快速定位规则、理解分支并据此实现和测试。完整性不能以重复、堆砌和文学化描述为代价。

## 1. 技术阅读路径

将复杂需求分成四层，读者按需深入：

| 层级 | 目的 | 必须包含 |
|---|---|---|
| L0 一页核心摘要 | 快速判断改什么、影响什么 | 目标、范围、核心改动、影响系统、阻塞问题 |
| L1 导航与全局关系 | 建立整体模型 | 功能清单、业务流程图、状态机、跨系统关系 |
| L2 可执行规则 | 支持开发与测试 | 主流程、分支、判定规则、字段、操作、权限、异常、验收 |
| L3 技术契约 | 支持落地和非功能设计 | 数据、接口、容量、性能、安全、可用性、监控和兼容性 |

上层只给结论和入口，不复制下层全部细节。每项核心改动必须能通过稳定编号定位到详细规则、图、原型和验收标准。

## 2. 控制信息密度

- 一段只表达一个核心规则；多个独立条件拆成列表、判定表或状态表。
- 不重复解释同一背景、字段或规则；使用稳定编号交叉引用。
- 将例外、补充和历史说明放入对应规则附近，避免散落在文末或批注中。
- 核心规则不得埋在长段落、截图批注或会议记录中。
- 删除不能改变实现和验收结论的修饰性文字。
- 详情很多时采用摘要加引用，不把所有附录内容塞进正文。

## 3. 统一术语

为可能产生歧义的业务对象、时间、状态、金额、角色和系统建立术语表。

| 标准术语 | 定义 | 禁止混用或历史别名 | 对应字段 / 状态 | 依据 |
|---|---|---|---|---|
| | | | | |

全文遵守“一概念一名称、一名称一含义”。必须兼容历史名称时，在术语表中建立一次映射，正文统一使用标准术语。发现“服务时间 / 官方服务时间”等疑似同义词时，先确认是否同一概念，不得自行合并。

## 4. 选择表达方式

不要把“图表优于文字”机械理解为所有内容都要画图。选择能最小化歧义和阅读成本的表达方式：

| 信息类型 | 推荐表达 | 不应只用 |
|---|---|---|
| 单一事实或简短约束 | 短句或列表 | 大表格 |
| 多字段、重复属性、映射关系 | 表格 | 连续长段落 |
| 跨角色、跨部门流程 | 泳道图或带角色的流程图 | 无主语流程描述 |
| 对象生命周期和状态流转 | 状态机图加状态流转表 | 仅列状态名称 |
| 多条件组合、优先级、互斥规则 | 判定表或判定树 | “如果……又如果……”长段落 |
| 接口调用、异步回调、两套账 | 时序图加接口规则表 | 仅画箭头 |
| 页面层级和跳转 | 原型图、页面截图或层级图 | 只写“进入下一页” |

图用于理解关系，表用于精确定义，文字用于解释目的和例外。三者互补，任何图都不能替代字段、条件、失败处理和验收规则。

复杂判定至少列出：条件组合、优先级、命中结果、是否继续判断、互斥范围和兜底结果。

## 5. 覆盖完整场景

每项功能至少扫描以下场景，只写与本需求相关的内容：

| 类型 | 检查内容 |
|---|---|
| 主流程 | 正常输入、正常状态、成功结果 |
| 业务分支 | 不同角色、产品、状态、金额、渠道和配置 |
| 数据边界 | 空值、零值、最大值、重复、过期、非法和缺失数据 |
| 网络与依赖 | 弱网、超时、断网、回调丢失、外部系统不可用 |
| 并发与幂等 | 重复点击、并发编辑、重复回调、重复任务 |
| 权限 | 页面不可见、按钮不可用、数据越权和权限中途变化 |
| 失败恢复 | 部分成功、重试、回滚、补偿、人工恢复和历史记录 |
| 兼容性 | 历史数据、进行中记录、旧版本和多端差异 |

异常场景必须说明系统状态、数据结果、用户提示、可执行动作、日志或告警，以及恢复后是否允许继续。

## 6. 评审与答疑

PRD 是协作契约，不是单向投递物。

以下需求在定稿前必须留下评审记录：跨系统、跨团队、状态复杂、资金账务、安全合规、数据迁移或高风险业务改动。简单需求可以异步评审，但同样需要明确确认人和结论。

评审前：

- 提前提供一页摘要、流程或状态图、原型和阻塞问题。
- 明确需要产品、业务、技术、测试、设计和数据中的哪些角色参加。
- 将待决策项整理成可选择的问题，不在会上临时翻找长文档。

评审后：

- 记录结论、否决方案、影响范围、负责人和截止时间。
- 为未解决问题指定答疑负责人和回复渠道。
- 更新 PRD、原型、UI、接口和验收标准，并高亮重要改动。
- 向受影响的开发、测试、设计、业务或运营人员说明变更编号和影响。

## 7. 文档与原型同步

| 交付物 | 必须一致的内容 | 责任人 | 当前版本 / 日期 | 状态 |
|---|---|---|---|---|
| PRD | 规则、字段、状态、异常、验收 | | | |
| 原型 | 页面结构、交互、跳转、状态 | | | |
| UI 设计 | 文案、视觉状态、控件 | | | |
| 接口文档 | 字段、方向、时机、返回、错误 | | | |
| 测试用例 | 主流程、分支、异常和边界 | | | |

任一交付物修改时，检查其他交付物是否受影响。存在版本不一致、批注未合并或原型已改而正文未改时，PRD 必须标记为未同步，不能认定完成。

## 8. 完成条件

复杂 PRD 只有同时满足以下条件才可标记完成：

- 一页摘要能定位核心改动和影响范围；
- 术语唯一且有定义；
- 复杂逻辑采用合适的图、表或判定结构；
- 主流程、分支、异常和边界足以直接实现和测试；
- PRD、原型、UI、接口和测试口径保持一致；
- 评审结论、待确认项和答疑负责人清晰；
- 重要变更已高亮并通知相关角色。
