# PRD 编写与校验

## 用途

编写业务用户看得懂，人、开发、测试和 AI 能一致执行的 PRD。严格区分已确认事实、业务决策、暂时假设和待确认问题。

文档厚度不代表有效性。复杂需求必须完整，但正文应让技术人员快速定位核心改动、可执行规则、边界、异常和验收标准；不能用冗长描述代替清晰逻辑。

## 技术可读性门禁

编写、校验或大幅修改 PRD 时，必须读取 [references/clarity-and-alignment.md](references/clarity-and-alignment.md)。如果存在以下任一问题，不得认定 PRD 完成：

- 核心规则淹没在长段落或重复内容中；
- 同一概念使用不同术语，或者术语没有定义；
- 复杂流程、状态或条件组合只能依靠读者脑补；
- 只覆盖正常流程，缺少适用的异常和边界；
- PRD、原型、UI、接口或测试口径不一致；
- 复杂或跨团队需求没有评审结论、待确认项或答疑负责人。

发现这些问题时，先简化结构、补充合适图表和协同记录，不要继续堆砌正文。

## 硬门禁：先确认现有逻辑

修改现有系统时，在相关现有逻辑确认前，不得编写、修改或认定依赖这些逻辑的需求已经完整。

必须确认当前的：

- 系统、模块、页面、入口、用户、角色和权限；
- 流程、状态、流转、操作人、触发点、触发条件和副作用；
- 字段、取值来源、计算规则、操作、校验和异常处理；
- 后台规则、系统集成、通知和操作留痕。

如果缺少直接影响本次需求的现有逻辑：

1. 停止输出依赖该逻辑的最终需求。
2. 建立**现状待确认清单**。
3. 每次只问一个影响最大的明确问题。
4. 存在多种理解时，列出不同理解并让用户选择或纠正。
5. 持续追问，直到相关现有逻辑得到明确确认。
6. 记录确认依据：用户说明、现有 PRD、截图、系统查看结果、代码或技术文档。

截图只能证明可见内容，不能证明隐藏的规则、权限、计算、触发条件或副作用。目标方案也不能反向证明现有逻辑。不得用“按常规系统处理”等假设绕过门禁。

## 选择工作模式

| 模式 | 输出目标 |
|---|---|
| 新建 | 确认业务故事和关键决策后编写新 PRD |
| 校验 | 输出已确认内容、缺失项、歧义、矛盾、风险和待确认问题 |
| 修改 | 确认现有逻辑和目标逻辑后，只做必要修改 |

真正从零建设且没有替代对象的新功能不适用现状门禁。如果“新功能”复用或替代现有页面、流程、接口、字段或规则，相关部分仍需执行现状门禁。

## 选择输出侧重点

| 需求类型 | 重点内容 |
|---|---|
| 通用业务系统 | 业务场景、流程、状态、页面、规则、异常和验收 |
| 管理后台、资金或接口类 | 字段、筛选、操作、权限、接口时机、成功条件、回调、账务和对账 |
| 移动端或 C 端 | 入口、触发、路径、页面状态、完整文案、脱敏、复制和异常引导 |

需求同时涉及多端时，按端分别描述并补充端到端流程。涉及资金、接口或移动端时，必须读取 [references/scenario-patterns.md](references/scenario-patterns.md)，只使用匹配的场景模板。

用户要求禅道式或简版需求时，可以压缩背景和目标，但不能省略现状确认、关键规则、异常、权限和验收事实。

## 处理文档格式

- 本地 Markdown 或纯文本：直接读取和编辑。
- Word：使用当前 Agent 可用的 Word 或文档读写能力；修改后进行渲染和视觉检查。当前 Agent 不具备该能力时，请用户提供可读取的导出文件，或只输出明确的修改方案，不得假装已经修改原文档。
- 飞书 Docx 或 Wiki 文档：使用当前 Agent 可用的飞书/Lark 文档连接器。没有连接器或权限时，请用户导出文档或补充正文，不得根据链接标题猜测内容。
- 截图、原型图或 UI 图：先查看图片，分配 `UI-xx` 编号，并与对应需求关联。

无法读取材料时，明确请求补充，不得凭记忆重建。修改现有文档时保留无关章节的结构和格式。

## 编号与可读性

- 默认编号层级使用 `1、` → `1）` → `⑴` → `①`，同一层级不得混用其他形式。
- 使用 `1）2）3）` 或 `⑴⑵⑶` 时，每个编号项必须独占一行，禁止挤成一段。
- 长子项另起一行并缩进；多个并列条件用分号分隔，避免一整段连续堆叠。
- 完整通知文案、字段值或用户可见提示使用独立引用块展示，不得只写“参考某文案”。

## 重要改动展示规则

对会改变业务流程、状态、字段取值、操作权限、系统规则、数据结果或验收结论的重要改动，必须使用**红色加粗**样式突出显示。

- Markdown：使用 `<span style="color:#FF0000"><strong>重要改动内容</strong></span>`。
- Word、飞书文档：对重要改动内容设置实际的红色字体和粗体样式。
- 只标记发生变化的关键内容，不得把整段、整页或未变化内容全部标红。
- “现有逻辑”和“新逻辑”同时出现时，优先标红加粗新逻辑及关键差异。

## 管理后台或内部管理系统页面写法

如果 PRD 或需求功能属于管理后台、运营后台或内部管理系统，每个适用页面必须按以下顺序说明，并写清每一项的取值、条件、权限、触发结果和异常处理：

1. 列表页字段及逻辑。
2. 筛选项及逻辑。
3. 操作按钮及逻辑。
4. 权限及逻辑。
5. 分页。
6. 其他页面元素，例如排序、批量选择、空状态、加载状态、异常状态和跳转行为。

编写或修改这类页面前，读取 [references/prd-template.md](references/prd-template.md) 中的管理后台页面模板；校验时检查是否缺项。

## 配图与图文一致性

以下内容存在多节点、分支或跨系统交互时必须配图：

- 跨系统或端到端流程；
- 非顺序状态流转；
- 接口调用、异步回调、两套账或资金划付时序；
- 菜单、Tab、卡片和按钮层级；
- 批量任务、跑批、消息或异步处理路径。

流程图、状态机、时序图或系统截图必须与正文配套，不得用图替代规则。每张图后必须增加 `> 图说：`，说明起点、关键判断、结果和异常分支。完成前交叉检查图与正文中的状态值、字段名、角色、接口方向和触发条件；任一方修改时同步修改另一方。

## 工作流程

1. 反讲并确认：**【业务用户】**在**【业务情境】**下执行**【操作】**，获得**【结果】**并避免**【风险】**。
2. 确认目标、范围、非目标、成功标准和受影响业务对象。
3. 读取 [references/clarity-and-alignment.md](references/clarity-and-alignment.md)，规划一页摘要、术语、图表、异常覆盖和评审方式。
4. 分开记录已确认事实、业务决策、暂时假设和待确认问题，禁止把假设静默写成事实。
5. 按产品边界，业务流程、状态、数据和权限，页面、字段和规则，最后体验和报表的顺序解决决策。优先给出带影响说明的 A/B 选项。
6. 涉及现有系统时，先通过现状确认门禁，再编写依赖现状的内容。
7. 选择输出侧重点；匹配资金、接口或移动端场景时读取 [references/scenario-patterns.md](references/scenario-patterns.md)。
8. 读取 [references/prd-template.md](references/prd-template.md)，新建或最小修改 PRD。使用 `SC-xx`、`PAGE-xxx`、`F-xx`、`FLOW-xx`、`ST-xx`、`RULE-xx`、`UI-xx`、`AC-xx`、`D-xx` 等稳定编号。
9. 使用**现有逻辑 → 新逻辑 → 影响范围**描述改动。无页面逻辑使用**现有规则 → 新规则 → 触发点 → 异常处理**描述。
10. 按重要改动展示规则标红加粗关键变化。
11. 读取 [references/review-checklist.md](references/review-checklist.md)，逐项检查所有适用内容。

## 完成标准

只有满足以下条件，才能认定 PRD 完成：业务用户能够理解业务流程；技术人员能够从一页摘要定位核心规则；术语统一；字段有明确取值规则；状态流转有操作人、触发点、条件、结果和副作用；现有逻辑与新逻辑明确区分；后台规则和异常处理完整；视觉材料有图说且与文字需求一致；编号清晰且每项独占一行；验收标准能够判断通过或失败；PRD、原型、UI、接口和测试口径保持同步；复杂需求已经评审并明确答疑负责人；重要改动已经红色加粗；管理后台页面已经按规定顺序说明完整。

如果仍有直接影响需求的现状问题，必须标记为**因现状未确认而阻塞**，只能交付部分草稿或校验报告，并继续每次追问一个问题。不得宣称 PRD 已完成。
